# Auction Investment Pipeline — Orchestrator

법원경매 물건을 **수집 → 권리분석 → 시세평가 → 수익성계산 → 투자등급 랭킹**의 5단계 파이프라인으로 분석한다.
5개 전문 에이전트를 순차/병렬로 조율하며, 데이터는 `_workspace/` 파일로 전달한다.

## 실행 모드

**하이브리드**: 수집(1단계)은 단일 에이전트, 권리분석·시세평가·수익성계산(2~4단계)은 물건별 **fan-out 서브 에이전트**(병렬), 랭킹(5단계)은 집계 단일 에이전트.
모든 `Agent` 호출은 `model: "opus"`, 물건별 병렬은 `run_in_background: true`.

## 데이터 계약 (`_workspace/`)

| 파일 | 생성 단계 | 내용 |
|------|----------|------|
| `01_listings.csv` | 수집 | 사건번호·소재지·물건종류·감정가·최저가·매각기일·유찰횟수 |
| `02_rights_{사건번호}.json` | 권리분석 | 말소기준권리·인수권리·인수금액·임차인현황·명도난이도(1~5) |
| `03_valuation_{사건번호}.json` | 시세평가 | 인근실거래중앙값·평당가·예상낙찰가·시세대비저감율 |
| `04_profit_{사건번호}.json` | 수익성 | 총투자금·예상수익·ROI·시나리오(보수/기본/낙관) |
| `05_ranking.json` / `05_report.html` | 랭킹 | 종합등급(S~D)·순위·근거·최종 리포트 |

사건번호는 파일명 안전 형식으로 정규화한다(예: `2024타경12345` → `2024_12345`).

## 파이프라인 흐름

```
1. 수집        auction-collector      → 01_listings.csv
2. 권리분석    rights-analyzer          ┐ 물건별 fan-out (병렬)
3. 시세평가    market-valuator          │ 각 사건번호에 대해
4. 수익성      profitability-calculator ┘ 2→3→4 순으로 물건별 파이프라인
5. 랭킹/리포트 investment-ranker      → 05_ranking.json + 05_report.html
```

**중요:** 2~4단계는 물건별로 독립적이므로 **물건 단위 파이프라인**으로 fan-out한다(물건 A가 시세평가 중일 때 물건 B는 권리분석 가능). 전 물건이 4단계를 마친 뒤 5단계로 barrier.

## 재사용 스킬 (중복 생성 금지)

각 에이전트는 아래 기존 글로벌 스킬을 사용한다. 새로 만들지 않는다:
- 수집 → `court-auction-scraper`
- 권리분석/수익성 근거 → `npl-analyzer`
- 시세평가 → `realprice-flow`, `apt-value`
- 랭킹/리포트 → `auction-investment-ranker` (이 하네스 전용 신규 스킬)

## 단계별 에이전트 호출

1. **수집**: `auction-collector`에게 지역/조건 전달 → `01_listings.csv`. 물건 0건이면 조건 완화 제안 후 중단.
2. **물건별 fan-out**: `01_listings.csv`의 각 행에 대해 `rights-analyzer` → `market-valuator` → `profitability-calculator`를 물건 파이프라인으로 실행. 상한: 1회 최대 30건(초과 시 최저가 오름차순 상위 30건, 나머지는 리포트에 "미분석"으로 명시).
3. **랭킹**: 전 물건의 `04_profit_*.json`을 모아 `investment-ranker`가 종합등급·순위·`05_report.html` 생성.

## 에러 처리 / 폴백

- 특정 물건의 PDF 명세서 없음 → 해당 물건 `02_rights`에 `"data_gap": true` 표기, 등급 산정 시 감점하고 리포트에 사유 명시(전체 중단 금지).
- 인근 실거래 부족 → `market-valuator`가 반경/기간 확대, 그래도 없으면 감정가 기반 추정 + 신뢰도 하향.
- 스크래핑 차단/실패 → `court-auction-scraper`의 스텔스 모드 지침을 따르고, 2회 실패 시 사용자에게 보고 후 중단.

## 테스트 시나리오

- **정상 흐름**: "서울 강남구 아파트 경매 물건 분석해줘" → 수집 N건 → 물건별 병렬 분석 → S~D 등급 리포트.
- **에러 흐름**: 수집 0건 → 조건 완화(지역 확대/물건종류 추가) 제안 후 재수집 또는 중단.
